# Структуруйте своє рішення за компонентами

<br/><br/>

### Пояснення за один абзац

Для застосунків середнього розміру і вище *немодульні* моноліти справді погані — мати одне велике програмне забезпечення зі "спагетті" залежностей просто складно осягнути. Найкращим рішенням є розробка меншого програмного забезпечення: розділіть весь стек на самодостатні компоненти, які не надають спільний доступ до файлів іншим, кожен є автономним логічним застосунком (наприклад, має власний API, сервіс, доступ до даних, тести тощо), щоб адаптація до нього та зміна коду була набагато легшою, ніж робота з усією системою. Дехто може назвати цю архітектуру "мікросервісною" — важливо розуміти, що мікросервіси — це не специфікація, якої ви повинні дотримуватися, а радше набір принципів. Ви можете прийняти багато принципів у повноцінну архітектуру мікросервісів або прийняти лише деякі. Щонайменше, що вам слід зробити, це створити базові межі між компонентами, призначити папку або репозиторій у корені вашої системи для кожного бізнес-компонента та зробити його самодостатнім. Іншим компонентам дозволено використовувати його функціональність лише через публічний інтерфейс або API. Це основа для збереження простоти ваших компонентів, уникнення пекла залежностей і прокладання шляху до повномасштабних мікросервісів у майбутньому, коли ваш застосунок розширяться.

### Цитата з блогу: "Масштабування вимагає масштабування всієї програми"

З блогу [MartinFowler.com](https://martinfowler.com/articles/microservices.html)

> Монолітні програми можуть бути успішними, але все більше людей відчувають розчарування в них, особливо тому, що все більше програм розгортається в хмарі. Цикли змін пов'язані разом — зміна, внесена до невеликої частини програми, вимагає перебудови та розгортання всього моноліту. З часом часто стає важко підтримувати хорошу модульну структуру, що ускладнює збереження змін, які мають впливати лише на один модуль у цьому модулі. Масштабування вимагає масштабування всієї програми, а не її частин, які потребують більших ресурсів.

### Цитата з блогу: "То що кричить архітектура вашої програми?"

 З блогу [uncle-bob](https://blog.cleancoder.com/uncle-bob/2011/09/30/Screaming-Architecture.html) 

> ...якби ви дивилися на архітектуру бібліотеки, ви б, швидше за все, побачили парадний вхід, зону для реєстраторів, зони для читання, невеликі конференц-зали та галерею за галереєю, здатні вмістити книжкові полиці для всіх книг в бібліотеці. Ця архітектура кричала б: Бібліотека.<br/>

Отже, про що кричить архітектура вашої програми? Коли ви дивитесь на структуру каталогів верхнього рівня та вихідні файли в пакеті найвищого рівня; вони кричать: система охорони здоров'я, чи система бухгалтерського обліку, чи система управління запасами? Або вони кричать: Rails, Spring/Hibernate або ASP?.

### Добре: Структуруйте своє рішення за допомогою автономних компонентів

```bash
my-system
├─ apps (components)
│  ├─ orders
│  │ ├─ package.json
│  │ ├─ api
│  │ ├─ domain
│  │ ├─ data-access
│  ├─ users
│  ├─ payments
├─ libraries (generic cross-component functionality)
│  ├─ logger
│  ├─ authenticator
```

### Погано: Групуйте файли за технічною роллю

```bash
│  ├─ payment-service.js
├─ models

│  ├─ order-model.js
│  ├─ payment-model.js
```
