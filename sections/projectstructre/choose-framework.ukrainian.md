# Враховуйте всі наслідки при виборі основного фреймворку

<br/><br/>

### Рекомендовані фреймворки: Переваги та недоліки

На відміну від інших рішень, вибір основного фреймворку визначає стратегічні фактори, такі як стиль розробки та ймовірність того, що команда зіткнеться з труднощами. Ми вважаємо, що популярність фреймворку є найважливішим критерієм, тому зосереджуємо увагу на 4 найпопулярніших фреймворках за кількістю завантажень та зірок на GitHub. Нижче наведено переваги та недоліки кожного фреймворку, а також рекомендації щодо вибору відповідного фреймворку для різних типів застосунків.

**express.js**

Переваги: Неперевершена популярність; величезна екосистема розширень та middleware; простий у вивченні та використанні; знайомий майже кожному Node.js розробнику; безліч статей та відео спільноти базуються на express

Недоліки: Охоплює лише малу частину типових потреб застосунку — по суті, це веб-сервер, який викликає функцію застосунку для кожного URL. Вибір express означає, що низка аспектів застосунку залишаться без покриття; застаріла механіка — відсутня нативна підтримка async-await; майже не підтримується та не оновлюється; повільніший за інших

**Nest.js**

Переваги: Більше можливостей "з коробки" ніж будь-який інший варіант — охоплює багато аспектів застосунку, включаючи черги повідомлень, заплановані задачі тощо; ООП-стиль є перевагою для команд, які цінують цей підхід до проєктування; чудова документація; добре підтримується; висока популярність із активною спільнотою

Недоліки: Високорівневі абстракції, які затьмарюють вбудовані конвенції Node.js; включення багатьох функцій, інтенсивне використання TypeScript та посилання на складні патерни можуть підштовхнути команди до надмірної складності; крутіша крива навчання через низку унікальних концепцій (наприклад, interceptors, guards, modules тощо); висока категоричність

**Fastify**

Переваги: Відносно простий та легкий; переважно базується на стандартах Node.js/JavaScript; відносно пологий поріг входу; офіційні плагіни охоплюють багато аспектів застосунку, хоча й не так багато як Nest.js

Недоліки: Молодший за інших та ще не такий популярний; менша екосистема порівняно з express та Nest.js

**Koa**

Переваги: У порівнянні з express: він простіший та легший; сучасний API з підтримкою async/await; краща продуктивність

Недоліки: Охоплює лише малу частину типових потреб застосунку — низка аспектів залишається без покриття; не такий популярний як express та Nest.js

### Короткий посібник з вибору

**Віддавайте перевагу express.js, коли** — в команді є досвідчений архітектор _та_ є потреба контролювати дрібні деталі. За таких обставин Koa також є надійним варіантом із більш сучасним API ніж express, але значно меншою екосистемою

**Віддавайте перевагу Fastify, коли —** застосунок складається з компонентів/мікросервісів розумного розміру (тобто, не величезний моноліт); для команд з надійними знаннями JavaScript та Node.js; коли бажано дотримуватися наративів та духу Node.js

**Віддавайте перевагу Nest.js, коли** — бажано проєктувати та писати код в ООП-стилі; коли команда має великий досвід роботи з Java/Spring/Angular або подібними технологіями; для застосунків великого розміру, які неможливо розбити (тобто моноліт) на автономні компоненти; для команди, якій бракує фундаментальних навичок JavaScript/Node.js (не виключно, це ще один фактор для розгляду); коли накладні витрати на прийняття рішень мають бути мінімізовані; коли час до першого релізу є критичним фактором
