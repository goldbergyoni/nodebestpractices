### Розділіть код компонента на 3 рівні

Корінь кожного компонента повинен містити 3 папки, які представляють загальні аспекти та етапи кожної транзакції:

```bash
my-system
├─ apps (components)
│  ├─ component-a
   │  ├─ entry-points
   │  │  ├─ api # контролер розміщується тут
   │  │  ├─ message-queue # споживач повідомлень розміщується тут
   │  ├─ domain # функції та потоки: DTO, сервіси, логіка
   │  ├─ data-access # виклики до БД без ORM
```

**- Entry-points (точки входу) -** Тут починаються запити та потоки, чи то REST API, Graph, черга повідомлень, заплановані завдання або будь-які інші _двері_ до застосунку. Відповідальність цього рівня досить мінімальна — адаптувати payload (наприклад, JSON) до формату застосунку, включаючи першу валідацію, викликати рівень логіки/домену та повернути відповідь. Зазвичай це досягається кількома рядками коду. Багато хто використовує термін "контролер" для цього типу коду, хоча технічно це просто адаптер

**- Domain (домен) -** Тут живуть потоки, логіка та дані застосунку. Цей рівень приймає протокол-агностичний payload, звичайний JavaScript об'єкт, і повертає такий само. Технічно він містить загальні об'єкти коду, такі як сервіси, dto/сутності та клієнти, які викликають зовнішні сервіси. Він також зазвичай викликає рівень доступу до даних для отримання або збереження інформації

**- Data-access (доступ до даних) -** Тут застосунок зберігає код, який взаємодіє з БД. В ідеалі він повинен надавати інтерфейс, який повертає/отримує звичайні JavaScript об'єкти, що не залежать від БД (також відомий як repository-pattern). Цей рівень включає допоміжні утиліти для БД, такі як query builders, ORM, драйвери БД та інші бібліотеки реалізації

**У чому користь? -** Маючи гнучку інфраструктуру, яка дозволяє швидко додавати більше викликів API та запитів до БД, розробник може кодувати функцію швидше, зосереджуючись на папці domain. Іншими словами, менше часу витрачається на технічні активності і більше на активності з доданою вартістю. Це всюдисуща риса, яка лежить в основі більшості архітектур програмного забезпечення, таких як DDD, hexagonal, clean-architecture та інші. Крім того, коли рівень домену не знає про жодний edge-протокол, він може обслуговувати будь-якого клієнта, а не лише HTTP-виклики

**Чому не MVC або clean architecture? -** 3-рівневий патерн забезпечує чудовий баланс між досягненням мети розділення та збереженням простоти структури. Йому також бракує абстракцій — кожен рівень представляє реальний фізичний рівень, який відвідає кожен запит. З іншого боку, MVC — це спрощений патерн, де літери VC представляють лише кілька рядків коду, а літера M означає все інше. Clean architecture — це архітектура з високим рівнем абстракцій, яка може досягти ще більшого розділення, але ціна непропорційно вища через підвищену складність
