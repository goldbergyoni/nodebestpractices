# Делегуйте все можливе (наприклад, статичний контент, gzip) реверсному проксі

<br/><br/>

### Пояснення за один абзац

Дуже спокусливо сліпо копіювати Express і використовувати його багатий middleware для мережевих завдань, таких як обслуговування статичних файлів, gzip-кодування, обмеження запитів, SSL-термінація тощо. Це вбивство продуктивності через його однопотокову модель, яка тримає CPU зайнятим протягом тривалих періодів (пам'ятайте, модель виконання Node оптимізована для коротких завдань або завдань, пов'язаних з асинхронним IO). Кращий підхід — використовувати інструмент, який спеціалізується на мережевих завданнях — найпопулярніші nginx та HAproxy, які також використовуються найбільшими хмарними провайдерами для полегшення вхідного навантаження на node.js процеси.

<br/><br/>

### Приклад конфігурації Nginx – Використання nginx для стиснення відповідей сервера

```nginx
# налаштування gzip стиснення
gzip on;
gzip_comp_level 6;
gzip_vary on;

# налаштування upstream
upstream myApplication {
    server 127.0.0.1:3000;
    server 127.0.0.1:3001;
    keepalive 64;
}

# визначення веб-сервера
server {
    # налаштування сервера з ssl та сторінками помилок
    listen 80;
    listen 443 ssl;
    ssl_certificate /some/location/sillyfacesociety.com.bundle.crt;
    error_page 502 /errors/502.html;

    # обробка статичного контенту
    location ~ ^/(images/|img/|javascript/|js/|css/|stylesheets/|flash/|media/|static/|robots.txt|humans.txt|favicon.ico) {
    root /usr/local/silly_face_society/node/public;
    access_log off;
    expires max;
}
```

### Що кажуть інші блогери

* З блогу [Mubaloo](http://mubaloo.com/best-practices-deploying-node-js-applications):
> …Дуже легко потрапити в цю пастку – Ви бачите пакет на кшталт Express і думаєте "Чудово! Почнемо" – ви кодуєте і отримуєте застосунок, який робить те, що ви хочете. Це чудово і, чесно кажучи, ви виграли більшу частину битви. Однак ви програєте війну, якщо завантажите свій застосунок на сервер і змусите його слухати ваш HTTP-порт, тому що ви забули дуже важливу річ: Node — це не веб-сервер. **Як тільки будь-який обсяг трафіку почне потрапляти у ваш застосунок, ви помітите, що все починає йти не так: з'єднання обриваються, ресурси перестають обслуговуватися або, в найгіршому випадку, ваш сервер падає. Те, що ви робите — це намагаєтесь змусити Node робити всі складні речі, які перевірений веб-сервер робить дуже добре. Навіщо винаходити велосипед?**
> **Це лише для одного запиту, для одного зображення і, враховуючи, що це пам'ять, яку ваш застосунок міг би використовувати для важливих речей, таких як читання бази даних або обробка складної логіки; навіщо калічити свій застосунок заради зручності?**

* З блогу [Argteam](http://blog.argteam.com/coding/hardening-node-js-for-production-part-2-using-nginx-to-avoid-node-js-load):
> Хоча express.js має вбудовану обробку статичних файлів через деякий connect middleware, ви ніколи не повинні її використовувати. **Nginx може набагато краще обробляти статичні файли та може запобігти засміченню наших node-процесів запитами на нединамічний контент**…
