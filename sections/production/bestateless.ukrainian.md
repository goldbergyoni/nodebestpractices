# Будьте без стану, "вбивайте" свої сервери майже щодня

<br/><br/>

### Пояснення за один абзац

Чи стикалися ви коли-небудь із серйозною виробничою проблемою, коли на одному сервері бракувало якоїсь частини конфігурації або даних? Ймовірно, це через якусь непотрібну залежність від локального ресурсу, який не є частиною розгортання. Багато успішних продуктів ставляться до серверів як до птаха фенікса — він помирає і періодично відроджується без будь-якої шкоди. Іншими словами, сервер — це просто апаратне забезпечення, яке виконує ваш код деякий час і замінюється після цього.
Цей підхід

- дозволяє масштабуватися шляхом динамічного додавання та видалення серверів без будь-яких побічних ефектів.
- спрощує обслуговування, оскільки звільняє наш розум від оцінки стану кожного сервера.

### Приклад коду: антипатерни

```javascript
// Типова помилка 1: збереження завантажених файлів локально на сервері
const multer = require('multer'); // express middleware для обробки multipart завантажень
const upload = multer({ dest: 'uploads/' });

app.post('/photos/upload', upload.array('photos', 12), (req, res, next) => {});

// Типова помилка 2: зберігання сесій автентифікації (passport) у локальному файлі або пам'яті
const FileStore = require('session-file-store')(session);
app.use(session({
    store: new FileStore(options),
    secret: 'keyboard cat'
}));

// Типова помилка 3: зберігання інформації в глобальному об'єкті
Global.someCacheLike.result = { somedata };
```

### Що кажуть інші блогери

З блогу [Martin Fowler](https://martinfowler.com/bliki/PhoenixServer.html):
> ...Одного дня у мене виникла фантазія про запуск сервісу сертифікації для операцій. Оцінка сертифікації полягала б у тому, що ми з колегою з'являлися б у корпоративному дата-центрі та атакували критичні виробничі сервери бейсбольною битою, бензопилою та водяним пістолетом. Оцінка базувалася б на тому, скільки часу знадобиться операційній команді, щоб усі застосунки знову запрацювали. Це може бути божевільною фантазією, але тут є зерно мудрості. Хоча вам слід відмовитися від бейсбольних бит, хорошою ідеєю є віртуально спалювати ваші сервери через регулярні проміжки часу. Сервер повинен бути як фенікс, регулярно відроджуючись з попелу...
