# Винесіть ваші frontend-ресурси з Node

<br/><br/>

### Пояснення за один абзац

У класичному веб-застосунку бекенд обслуговує фронтенд/графіку для браузера, дуже поширений підхід у світі Node — використовувати Express static middleware для потокової передачі статичних файлів клієнту. АЛЕ — Node не є типовим веб-застосунком, оскільки він використовує один потік, який не оптимізований для обслуговування багатьох файлів одночасно. Замість цього розгляньте можливість використання реверсного проксі (наприклад, nginx, HAProxy), хмарного сховища або CDN (наприклад, AWS S3, Azure Blob Storage тощо), які використовують багато оптимізацій для цього завдання та отримують набагато кращу пропускну здатність. Наприклад, спеціалізований middleware, такий як nginx, має прямі хуки між файловою системою та мережевою картою і використовує багатопотоковий підхід для мінімізації втручання між кількома запитами.

Ваше оптимальне рішення може мати одну з наступних форм:

1. Використання реверсного проксі — ваші статичні файли будуть розташовані прямо поруч з вашим Node-застосунком, лише запити до папки статичних файлів будуть обслуговуватися проксі, який стоїть перед вашим Node-застосунком, таким як nginx. При такому підході ваш Node-застосунок відповідає за розгортання статичних файлів, але не за їх обслуговування. Ваш колега з фронтенду полюбить цей підхід, оскільки він запобігає cross-origin-запитам з фронтенду.

2. Хмарне сховище — ваші статичні файли НЕ будуть частиною вмісту вашого Node-застосунку, вони будуть завантажені до сервісів, таких як AWS S3, Azure BlobStorage або інших подібних сервісів, які були створені для цієї місії. При такому підході ваш Node-застосунок не відповідає ні за розгортання статичних файлів, ні за їх обслуговування, отже, досягається повне розділення між Node та Frontend, які все одно обробляються різними командами.

### Приклад конфігурації: типова конфігурація nginx для обслуговування статичних файлів

```nginx
# налаштування gzip стиснення
gzip on;
keepalive 64;

# визначення веб-сервера
server {
listen 80;
listen 443 ssl;

# обробка статичного контенту
location ~ ^/(images/|img/|javascript/|js/|css/|stylesheets/|flash/|media/|static/|robots.txt|humans.txt|favicon.ico) {
root /usr/local/silly_face_society/node/public;
access_log off;
expires max;
}
```

### Що кажуть інші блогери

З блогу [StrongLoop](https://strongloop.com/strongblog/best-practices-for-express-in-production-part-two-performance-and-reliability/):

>…У розробці ви можете використовувати [res.sendFile()](http://expressjs.com/4x/api.html#res.sendFile) для обслуговування статичних файлів. Але не робіть цього в продакшені, тому що ця функція повинна читати з файлової системи для кожного запиту файлу, тому вона матиме значну затримку та вплине на загальну продуктивність застосунку. Зверніть увагу, що res.sendFile() не реалізований з системним викликом sendfile, який зробив би його набагато ефективнішим. Замість цього використовуйте serve-static middleware (або щось еквівалентне), яке оптимізоване для обслуговування файлів для Express-застосунків. Ще кращий варіант — використовувати реверсний проксі для обслуговування статичних файлів; див. Використання реверсного проксі для отримання додаткової інформації…
