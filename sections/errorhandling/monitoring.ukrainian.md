# Моніторинг

### Пояснення за один абзац

> На базовому рівні моніторинг означає, що ви можете *легко визначити, коли в продакшені трапляються погані речі. Наприклад, отримуючи сповіщення електронною поштою або Slack. Завдання полягає в тому, щоб вибрати правильний набір інструментів, які задовольнять ваші вимоги, не спустошуючи ваш гаманець. Можу запропонувати почати з визначення основного набору метрик, які необхідно відстежувати для забезпечення здорового стану — CPU, RAM сервера, RAM процесу Node (менше 1.4GB), кількість помилок за останню хвилину, кількість перезапусків процесу, середній час відповіді. Потім перегляньте деякі розширені функції, які вам можуть сподобатися, і додайте їх до списку бажань. Ось кілька прикладів розкішних функцій моніторингу: профілювання БД, крос-сервісне вимірювання (тобто вимірювання бізнес-транзакцій), інтеграція з фронтендом, надання необроблених даних для користувацьких BI-клієнтів, сповіщення Slack та багато іншого.

Досягнення розширених функцій вимагає тривалого налаштування або придбання комерційного продукту, такого як Datadog, NewRelic тощо. На жаль, досягнення навіть базових речей — це не прогулянка парком, оскільки деякі метрики пов'язані з апаратним забезпеченням (CPU), а інші живуть всередині процесу node (внутрішні помилки), тому всі прямолінійні інструменти вимагають додаткового налаштування. Наприклад, рішення для моніторингу хмарних провайдерів (наприклад, AWS CloudWatch, Google StackDriver) одразу повідомлять вам про апаратні метрики, але нічого про внутрішню поведінку додатку. З іншого боку, рішення на основі логів, такі як ElasticSearch, за замовчуванням не мають апаратного огляду. Рішення полягає в тому, щоб доповнити ваш вибір відсутніми метриками, наприклад, популярний вибір — надсилати логи додатку до Elastic stack і налаштувати додаткового агента (наприклад, Beat) для надання інформації, пов'язаної з апаратним забезпеченням, щоб отримати повну картину.
