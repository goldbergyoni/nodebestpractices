# Будьте обережні при роботі з дочірніми процесами

### Пояснення за один абзац

Як би не були чудові дочірні процеси, їх слід використовувати з обережністю. Передача користувацького вводу повинна бути санітизована, якщо її взагалі не уникати.
Небезпека несанітизованого вводу, що виконує логіку системного рівня, необмежена, починаючи від віддаленого виконання коду до розкриття конфіденційних системних даних і навіть втрати даних. Чек-лист підготовки може виглядати так:

- уникайте користувацького вводу в кожному випадку, інакше валідуйте і санітизуйте його
- обмежте привілеї батьківського та дочірнього процесів, використовуючи ідентифікатори користувача/групи
- запускайте ваш процес в ізольованому середовищі, щоб запобігти небажаним побічним ефектам, якщо інші підготовки не спрацюють

### Приклад коду: Небезпеки несанітизованого виконання дочірніх процесів

```javascript
const { exec } = require('child_process');

...

// як приклад, візьміть скрипт, який приймає два аргументи, один з яких є несанітизованим користувацьким вводом
exec('"/path/to/test file/someScript.sh" --someOption ' + input);

// -> уявіть, що може статися, якщо користувач просто введе щось на кшталт '&& rm -rf --no-preserve-root /'
// вас чекає неприємний сюрприз
```

### Додаткові ресурси

З [документації](https://nodejs.org/dist/latest-v8.x/docs/api/child_process.html#child_process_child_process_exec_command_options_callback) Node.js child process:

> Ніколи не передавайте несанітизований користувацький ввід до цієї функції. Будь-який ввід, що містить метасимволи оболонки, може бути використаний для запуску довільного виконання команд.
