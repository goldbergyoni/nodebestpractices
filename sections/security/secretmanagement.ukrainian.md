# Витягуйте секрети з конфігураційних файлів або використовуйте npm пакет, що їх шифрує

### Пояснення за один абзац

Найпоширеніший і безпечний спосіб надати додатку Node.js доступ до ключів і секретів — зберігати їх за допомогою змінних середовища в системі, де він запущений. Після встановлення до них можна отримати доступ з глобального об'єкта `process.env`.
Лакмусовий тест на те, чи правильно винесена вся конфігурація з коду додатку — чи може кодова база бути відкритою в будь-який момент без компрометації будь-яких облікових даних.

Для рідкісних ситуацій, коли секрети все ж потрібно зберігати в системі контролю версій, використання пакета, такого як [cryptr](https://www.npmjs.com/package/cryptr), дозволяє зберігати їх у зашифрованому вигляді, а не у вигляді простого тексту.

Існує різноманітність інструментів, які використовують git commit для аудиту комітів і повідомлень комітів на випадкові додавання секретів, такі як [git-secrets](https://github.com/awslabs/git-secrets).

### Приклад коду

Доступ до API ключа, збереженого в змінній середовища:

```javascript
    const azure = require('azure');

    const apiKey = process.env.AZURE_STORAGE_KEY;
    const blobService = azure.createBlobService(apiKey);
```

Використання `cryptr` для зберігання зашифрованого секрету:

```javascript
const Cryptr = require('cryptr');
const cryptr = new Cryptr(process.env.SECRET);
 
let accessToken = cryptr.decrypt('e74d7c0de21e72aaffc8f2eef2bdb7c1');
 
console.log(accessToken);  // виводить розшифрований рядок, який не зберігався в системі контролю версій
```

### Що кажуть інші блогери

> Env vars легко змінювати між розгортаннями без зміни будь-якого коду; на відміну від конфігураційних файлів, мала ймовірність того, що їх випадково закомітять до репозиторію коду; і на відміну від користувацьких конфігураційних файлів або інших механізмів конфігурації, таких як Java System Properties, вони є стандартом, незалежним від мови та ОС. [З: The 12 factor app](https://12factor.net/config)
