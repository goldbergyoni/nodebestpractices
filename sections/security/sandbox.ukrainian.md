# Запускайте небезпечний код у пісочниці

### Пояснення за один абзац

Як правило, слід запускати лише власні JavaScript файли. Теорії осторонь, реальні сценарії вимагають виконання JavaScript файлів, які передаються динамічно під час виконання. Наприклад, розгляньте динамічний фреймворк, такий як webpack, який приймає власні завантажувачі і виконує їх динамічно під час збірки. За наявності якогось шкідливого плагіна ми хочемо мінімізувати шкоду і, можливо, навіть дозволити потоку успішно завершитися — це вимагає запуску плагінів у середовищі пісочниці, яке повністю ізольоване з точки зору ресурсів, збоїв та інформації, якою ми ділимося з ним. Три основні варіанти можуть допомогти досягти цієї ізоляції:

- виділений дочірній процес — це забезпечує швидку інформаційну ізоляцію, але вимагає приборкання дочірнього процесу, обмеження часу його виконання та відновлення після помилок
- хмарний serverless фреймворк відповідає всім вимогам пісочниці, але розгортання та динамічний виклик FaaS функції — це не прогулянка парком
- деякі npm бібліотеки, такі як [sandbox](https://www.npmjs.com/package/sandbox) і [vm2](https://www.npmjs.com/package/vm2), дозволяють виконувати ізольований код в 1 рядок коду. Хоча цей останній варіант виграє в простоті, він забезпечує обмежений захист

### Приклад коду - Використання бібліотеки Sandbox для запуску коду в ізоляції

```javascript
const Sandbox = require('sandbox');
const s = new Sandbox();

s.run('lol)hai', (output) => {
  console.log(output);
  //output='Syntax error'
});

// Example 4 - Restricted code
s.run('process.platform', (output) => {
  console.log(output);
  //output=Null
});

// Example 5 - Infinite loop
s.run('while (true) {}', (output) => {
  console.log(output);
  //output='Timeout'
});
```
