# Запобігайте перевантаженню однопоточного виконання шкідливими RegEx

### Пояснення за один абзац

Ризик, притаманний використанню регулярних виразів, полягає в обчислювальних ресурсах, необхідних для парсингу тексту та зіставлення з заданим шаблоном. Для платформи Node.js, де домінує однопоточний event-loop, CPU-bound операція, така як розв'язання шаблону регулярного виразу, зробить додаток нереагуючим.
Уникайте RegEx, коли це можливо, або делегуйте завдання спеціалізованій бібліотеці, такій як [validator.js](https://github.com/chriso/validator.js) або [safe-regex](https://github.com/substack/safe-regex), щоб перевірити, чи є шаблон RegEx безпечним.

Деякі [приклади OWASP](https://www.owasp.org/index.php/Regular_expression_Denial_of_Service_-_ReDoS) вразливих шаблонів RegEx:
* (a|aa)+
* ([a-zA-Z]+)*

<br/><br/>

### Приклад коду – Валідація RegEx з експоненціальним часом і використання валідаторів замість RegEx

```javascript
const saferegex = require('safe-regex');
const emailRegex = /^([a-zA-Z0-9])(([\-.]|[_]+)?([a-zA-Z0-9]+))*(@){1}[a-z0-9]+[.]{1}(([a-z]{2,3})|([a-z]{2,3}[.]{1}[a-z]{2,3}))$/;

// повинно вивести false, тому що emailRegex вразливий до redos атак
console.log(saferegex(emailRegex));

// замість шаблону regex, використовуйте validator:
const validator = require('validator');
console.log(validator.isEmail('liran.tal@gmail.com'));
```

### Цитата з книги: "Вразливий регулярний вираз відомий як той, що застосовує повторення"

З книги [Essential Node.js Security](https://leanpub.com/nodejssecurity) від Liran Tal
> Часто програмісти використовують RegEx для валідації того, що ввід, отриманий від користувача, відповідає очікуваній умові. Вразливий регулярний вираз відомий як той, що застосовує повторення до повторюваної групи захоплення, і де рядок для зіставлення складається з суфікса дійсного відповідного шаблону плюс символів, які не відповідають групі захоплення.
