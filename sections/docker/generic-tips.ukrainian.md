# Загальні найкращі практики Docker для Node.js

Цей розділ загальних рекомендацій Docker містить найкращі практики, які стандартизовані для всіх мов програмування і не мають спеціальної інтерпретації для Node.js

## ![✔] Надавайте перевагу COPY над командою ADD

**TL;DR:** COPY безпечніший, оскільки він копіює лише локальні файли, тоді як ADD підтримує більш складні завантаження, такі як завантаження бінарних файлів з віддалених сайтів

## ![✔] Уникайте оновлення базової ОС

**TL;DR:** Оновлення локальних бінарних файлів під час збірки (наприклад, apt-get update) створює неузгоджені образи кожного разу, коли воно запускається, а також вимагає підвищених привілеїв. Замість цього використовуйте базові образи, які часто оновлюються

## ![✔] Класифікуйте образи за допомогою міток

**TL;DR:** Надання метаданих для кожного образу може допомогти фахівцям Ops належним чином обробляти його. Наприклад, включіть ім'я супроводжувача, дату збірки та іншу інформацію, яка може виявитися корисною, коли комусь потрібно проаналізувати образ

## ![✔] Використовуйте непривілейовані контейнери

**TL;DR:** Привілейований контейнер має ті самі дозволи та можливості, що й користувач root на хост-машині. Це рідко потрібно, і як правило, слід використовувати користувача 'node', який створюється в офіційних образах Node

## ![✔] Перевіряйте та верифікуйте кінцевий результат

**TL;DR:** Іноді легко не помітити побічні ефекти в процесі збірки, такі як витік секретів або непотрібні файли. Перевірка створеного образу за допомогою таких інструментів, як [Dive](https://github.com/wagoodman/dive), може легко допомогти виявити такі проблеми

## ![✔] Виконуйте перевірку цілісності

**TL;DR:** Під час завантаження базових або кінцевих образів мережа може бути введена в оману та перенаправлена на завантаження шкідливих образів. Ніщо в стандартному протоколі Docker не запобігає цьому, якщо не підписувати та не перевіряти вміст. [Docker Notary](https://docs.docker.com/notary/getting_started/) — один з інструментів для досягнення цього
