# Встановлюйте ліміти пам'яті за допомогою Docker та v8

<br/><br/>

### Пояснення за один абзац

Ліміт пам'яті повідомляє процесу/контейнеру максимально дозволене використання пам'яті — запит або використання понад це число вб'є процес (OOMKill). Застосування цього — чудова практика для забезпечення того, щоб один громадянин не випив весь сік сам і не залишив інші компоненти голодувати. Ліміти пам'яті також дозволяють середовищу виконання розміщувати контейнер у правильному екземплярі — розміщення контейнера, який споживає 500 МБ, в екземплярі з 300 МБ доступної пам'яті призведе до збоїв. Дві різні опції дозволяють налаштувати цей ліміт: прапорці V8 (--max-old-space-size) та Docker runtime, обидва абсолютно необхідні. Переконайтеся, що завжди налаштовуєте ліміти Docker runtime, оскільки він має набагато ширшу перспективу для прийняття правильних рішень щодо здоров'я: Враховуючи цей ліміт, середовище виконання знає, як масштабувати та створювати більше ресурсів. Воно також може прийняти продумане рішення про те, коли впасти — якщо контейнер має короткий сплеск у запиті пам'яті і хостинг-екземпляр здатний це підтримати, Docker дозволить контейнеру залишитися живим. Нарешті, з Docker експерти Ops можуть встановлювати різні продакшен-конфігурації пам'яті, які можуть бути враховані, такі як memory swap. Цього самого по собі буде недостатньо — без встановлення --max-old-space-size v8, JavaScript runtime не буде підштовхувати збирання сміття при наближенні до лімітів і також впаде при використанні лише 50-60% хостового середовища. Відповідно, встановіть ліміт v8 на рівні 75-100% від ліміту пам'яті Docker.

### Приклад коду – Ліміт пам'яті з Docker

<details>
<summary><strong>Bash</strong></summary>

```bash
docker run --memory 512m my-node-app
```

</details>

<br/><br/>

### Приклад коду – Ліміт пам'яті з Kubernetes та v8

<details>
<summary><strong>Kubernetes deployment yaml</strong></summary>

```yml
apiVersion: v1
kind: Pod
metadata:
  name: my-node-app
spec:
  containers:
  - name: my-node-app
    image: my-node-app
    resources:
      requests:
        memory: "400Mi"
      limits:
        memory: "500Mi"
    command: ["node index.js --max-old-space-size=350"]
```

</details>

### Документація Kubernetes: "Якщо ви не вказуєте ліміт пам'яті"

З [документації K8S](https://kubernetes.io/docs/tasks/configure-pod-container/assign-memory-resource/)

> Контейнер не має верхньої межі обсягу пам'яті, яку він використовує. Контейнер може використовувати всю пам'ять, доступну на Node, де він працює, що в свою чергу може викликати OOM Killer. Крім того, у випадку OOM Kill, контейнер без лімітів ресурсів матиме більший шанс бути вбитим.

<br/><br/>

### Документація Docker: "він викидає OOME і починає вбивати процеси"

З [офіційної документації Docker](https://docs.docker.com/config/containers/resource_constraints/)

> Важливо не дозволяти запущеному контейнеру споживати занадто багато пам'яті хост-машини. На Linux-хостах, якщо ядро виявляє, що недостатньо пам'яті для виконання важливих системних функцій, воно викидає OOME, або Out Of Memory Exception, і починає вбивати процеси для звільнення пам'яті.

<br/><br/>

### Документація Node.js: "V8 витратить більше часу на збирання сміття"

З [офіційної документації Node.js](https://nodejs.org/api/cli.html#cli_max_old_space_size_size_in_megabytes)

> Встановлює максимальний розмір пам'яті старої секції пам'яті V8. Коли споживання пам'яті наближається до ліміту, V8 витратить більше часу на збирання сміття, намагаючись звільнити невикористану пам'ять. На машині з 2 ГБ пам'яті розгляньте встановлення цього значення на 1536 (1.5 ГБ), щоб залишити трохи пам'яті для інших цілей та уникнути свопінгу.
